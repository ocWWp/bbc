# BBC security audit — CSO comprehensive pass

**Date:** 2026-05-18
**Auditor:** Claude (Opus 4.7) running the `cso --comprehensive` skill (OWASP Top 10, STRIDE, secrets archaeology, supply chain, LLM security, webhook integrity).
**Codebase commit:** `dbd2d64` (main post-PR #37).
**Scope:** application code (`apps/dashboard/`, `packages/store/`) — auth, server actions, API routes, LLM input flow, secrets in history, dependencies, and webhook integrity.

This is the answer to "do we ever do a code-level security review?" — yes, this one, and a prior pre-launch audit (`docs/internal-launch-audit.md`) plus multiple codex passes that already caught three real bugs (silent BYOK fallback, /welcome bypass scoping, the BYOK Buffer→bytea storage bug). What follows is the OWASP-style code-vuln pass that the pre-launch audit explicitly did NOT do — it was feature-honesty focused, not code-vuln focused.

## Verdict

**POSTURE: STRONG.** BBC is built much more carefully than the user feared.

Every place a competent attacker would probe — SQL injection, XSS, IDOR, command injection, open redirect, prompt injection, secrets in git, webhook spoofing — has an explicit defense in code, with comments that name the threat. The few real bugs that existed were caught by prior codex passes and fixed in PRs #20-25. This pass found **one MEDIUM** and **one LOW**, both surface-cleanup rather than exploitable.

## Findings

### M1 — Leftover dev-spike route still public on prod

- **Severity:** MEDIUM
- **Confidence:** 10/10 — file exists, docstring explicit, middleware confirms no auth, M2 shipped
- **Status:** VERIFIED
- **Phase:** 1 (Attack Surface Census)
- **Category:** Surface Cleanup
- **File:** `apps/dashboard/src/app/api/spike-v16/stream/route.ts`
- **What:** A no-auth SSE endpoint emits five `tick` events then closes. Documented at line 8-9: *"Delete this route after M2 ships and the production /api/home/turn endpoint replaces it as the canonical streaming surface."* M2 shipped in v1.6; we are now in v1.8-prep. Middleware at `apps/dashboard/src/middleware.ts:15` explicitly allows the prefix.
- **Exploit:** Anyone on the internet can `curl https://bigbraincompany.online/api/spike-v16/stream` and hold a 1-second SSE stream. No data leaks, no auth bypass — but the surface should not exist on prod. An attacker scanning for live routes will find it, and "leftover dev endpoints" is the kind of finding a real pentest would flag.
- **Impact:** Unnecessary attack surface. Tells an attacker the codebase has leftover dev hooks — useful intel for follow-up probing.
- **Fix:** Delete the route file, delete the matching `startsWith("/api/spike-v16")` line in middleware. Both removals belong in the same commit so middleware doesn't reference a non-existent prefix.

### L1 — Inconsistent shell-out pattern in `LocalQueueStore`

- **Severity:** LOW (informational)
- **Confidence:** 8/10 — pattern is right, just not consistent
- **Status:** VERIFIED, not currently exploitable
- **Phase:** 9 (A03 — Injection)
- **Category:** Defense in Depth
- **File:** `packages/store/src/local/queue.ts:166`, `:189` (acceptProposal, rejectProposal)
- **What:** `fileProposal` uses `execFile` with argv-style argument passing (no shell parsing) — the textbook safe pattern. `acceptProposal` and `rejectProposal` use the `exec` family with manual `shq` POSIX-quote escaping. Both inputs (`proposalId`, `actor`) are regex-validated upstream and the `shq` quoter is correct, so this is not exploitable today. But two patterns in the same file invite mistakes when the next contributor copy-pastes from the wrong example.
- **Exploit:** Not currently exploitable. To make it exploitable, someone would need to (a) remove or weaken the upstream regex AND (b) not notice that `acceptProposal` relies on it. The risk is *future regression*, not present harm.
- **Fix:** Convert both `acceptProposal` and `rejectProposal` to use `execFile` like `fileProposal`. ~10 lines, no behavior change.

## What's solid (explicit positives)

Worth recording so the next audit doesn't re-spend cycles here. Each item below was traced to specific code, not assumed.

| Concern | Status | Evidence |
|---|---|---|
| **SQL injection** | Near-zero surface | `grep` for raw SQL / `queryRaw` / `executeRaw` returns zero hits in app code. All DB access is via Supabase client; 207 `.from()`/`.rpc()` callsites, all parameterized. |
| **XSS via raw-HTML injection** | Zero in user-content paths | One match for the raw-HTML React API exists in `landing/_components/safe-render.tsx`, and that file explicitly **avoids** the dangerous API by parsing trusted tags into proper React elements. Source content is compile-time constants. |
| **Code exec (`eval`/`Function`)** | Zero | No matches. |
| **Command injection (`child_process` in dashboard)** | Zero | Dashboard code never imports `child_process`. The shell-out lives in `packages/store/src/local/queue.ts` and is regex-gated + `shq`-escaped (see L1 for style cleanup). |
| **Auth gates** | Every mutating server action gated | Every `actions.ts` file across `/studio/*`, `/settings/*`, `/welcome`, `/queue` calls `requireActor()` + `requireRole()` before touching data. Verified via grep across all 30 action files. |
| **IDOR via direct-fetch by UUID** | Mitigated | `/api/v1/brain/memories/[id]/route.ts:28` calls `getMemory(adminClient, resolved.tenant_id, id)` — explicit tenant scope even though `adminClient` bypasses RLS. Brain-api `getMemory` at `brain-api.ts:113-118` enforces `.eq("tenant_id", tenantId)` *and* role-based type allowlist. Returns 404 on cross-tenant attempt (doesn't leak existence). |
| **Open redirect** | Properly mitigated | `apps/dashboard/src/app/auth/callback/route.ts:11-14` blocks both `//evil.com` (protocol-relative) and `/\evil.com` (browser-quirk) forms. Comment explicitly names both attacks. |
| **Bearer-token REST shim** | Well-built | `lib/api-auth.ts`: token regex `/^Bearer\s+(bbc_[A-Za-z0-9_]+\.[a-f0-9]+)$/`, verified via SQL RPC `resolve_api_key`, scope-checked via `scopeAllows`, role-checked via `allowedTypesForRole`. Returns null on every failure (no enumeration leak). |
| **Webhook signature verification** | HMAC-SHA256 + replay window + body cap | `/api/v1/webhooks/[tenant]/[webhook_id]/route.ts`: `X-BBC-Signature: sha256=<hex>` over `${timestamp}.${body}`, 5-min replay window, 1MB body cap with bounded reader. Dead-letter queue on failure. |
| **LLM prompt injection** | User input never enters system prompt | System prompts (`memory/extractor/prompt.ts`) are compile-time constants. User input goes to the user-message position only — per OWASP/Snyk guidance and the `cso` skill's Phase 7 precedent #13, that's not prompt injection. |
| **PII pre-scrub before LLM call** | Done | `welcome/actions.ts:21-43` redacts OpenAI / Anthropic / AWS / GitHub PAT / URL-passwords from user input *before* calling the extractor. Defense against accidental key paste, not malicious prompt injection. |
| **Secrets in git history** | Clean | `git log --all -G '(AKIA|sk-ant-|sk-proj-|ghp_|xoxb-)'` returns only documentation/placeholder strings. No `.env` files ever tracked. `.gitignore` covers `.env` and `.env.*`. |
| **Edge layer** | WAF + rate limits + bot baseline documented | See `docs/security/cloudflare-waf.md` and `launch-checklist.md`. Pre-DNS-cutover checklist exists. |
