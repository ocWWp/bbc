## What's new

### `/home` is now conversational

The widget dashboard is replaced by a chat surface. Type a question or describe what you want watched; BBC answers with text + action cards + citation chips, all streamed via SSE. Each tenant has one rolling session; turns persist across page refreshes with status='in_progress' / 'completed' / 'aborted' / 'failed' so an interrupted stream renders as "interrupted" instead of a partial-looking message.

The agent's reply uses a per-intent classifier (six intents: navigate, draft, watch, explain, meta, unclear) to pick a narrow tool subset. Grounding is verified post-LLM — claims that don't map to a retrieved memory id are stripped before render. Real Anthropic SDK wiring lands in a future release; v1.6 ships canned per-intent stubs that exercise the orchestration end-to-end.

### Loop 3: observers

The third BBC loop ("improve") is now wired: BBC watches a metric, files an anomaly as a proposal, and lets the user accept/reject like any other queue item. v1.6 includes:

- A PostHog adapter (read-only, daily-aggregate snapshots) — first capability class in the new tool-bundle-per-role pattern.
- Z-score anomaly detection with min-sample + cooldown gates.
- `POST /api/observer/run-now/:id` — operator+ gated, runs a check synchronously and files a proposal if the score crosses the threshold.
- `accept_proposal_observation()` RPC — atomic queue→memory promotion that creates the `memory_files` row only on accept (staged content stays in the queue until then).

### Three-step consent for new watches

"Watch my churn rate" no longer creates an active monitor in one click. Three discrete actions are required:

1. Agent emits a `watch_proposed` action card with the proposed metric spec.
2. **Set up this watch** → `/api/observer/signals/setup` creates a `disabled` signal row and validates the metric against the local catalog.
3. **Enable watching** → flips `enabled=true`; the signal joins the watching strip on /home.

Idempotent at the DB layer (unique partial index on `tenant_id + signal_type + metric` where not deleted), so concurrent setups can't land duplicate rows.

### Settings → Observers

New page at `/settings/observers`:

- Lists all enabled + paused watches with last-run time + status.
- Per-row actions: toggle (pauses or resumes), **Run check now** (synchronous), **Delete** (soft-delete; runs stay in audit log).
- `/settings/observers/[signalId]/runs` shows every run for that signal with outcome, token cost, and a deep link to the proposal in /queue.

### "Watching" strip on /home

Above the chat turns. Each enabled signal renders as a chip; clicking seeds the composer with "tell me about my <metric> watch" so the user can pivot into a focused conversation in one click.

### Atomic per-tenant quotas

`reserve_quota()` / `reconcile_quota()` ship as SECURITY DEFINER RPCs with hard-coded v1.6 caps:

| Cap | Default |
|---|---|
| `max_tokens_per_day` | 1,000,000 |
| `max_turns_per_day` | 1,000 |
| `max_runs_per_day` | 240 |
| `max_active_signals` | 10 |

Concurrency-safe under contention: 100 parallel reserves against 10 tokens of headroom yields exactly 10 successes / 90 `tokens_exceeded` / no overspend (verified in `test/rls/tenant_quotas.rls.test.ts`). Both RPCs require tenant membership; cross-tenant calls return `forbidden` / `not_found`.

### Mobile menu restored

The `<860px` AppNav burger was CSS-only since Phase P — the button was styled but never rendered. M5 wires the actual stateful button + popover panel so mobile users can reach Memory / Queue / Library / Settings.

## Breaking changes

None at the API boundary. Existing route handlers (`/api/queue/*`, `/settings/*`, `/studio/*`) are unchanged. The schema additions (7 new tables/triggers/functions across migrations 0045–0051) are additive.

## Deferred to v1.7

Surfaced during M5.2 codex review or implementation:

- **Real Anthropic SDK + GroundingVerifier tuning** — v1.6 ships stub LLM responses.
- **Voice review pass on real LLM samples** — requires the SDK wiring above first.
- **Per-user-per-hour cap** — `tenant_quotas` schema is tenant-keyed only; multi-user-per-tenant scenarios need a per-`(tenant, actor, hour_bucket)` counter.
- **Observer cron** — v1.6 runs are manually triggered. Cron + better detection lands in v1.7.
- **Operations_log version race** — `propose_observation` and the legacy accept RPCs compute `max(v)+1` without per-tenant serialization; concurrent observer runs for the same tenant can collide on the unique index. Benign under v1.6's single-user demo scenarios; needs proper sequencing in v1.7.
- **Pre-existing accept_proposal_observation queue race** — read-then-update without `FOR UPDATE` on the queue row. Fix: lock or status-conditional UPDATE.
- **Orphan reservation refund** — current behavior holds budget conservatively (a >5min orphan keeps its `estimated_tokens` even if the real LLM call comes back later with a smaller `actual_tokens`). v1.7 will reconcile.
