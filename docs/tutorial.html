1. Why BBC exists

Most teams accumulate decisions in Slack threads, Notion pages, and individual heads. The result is drift: agents can't reliably find your voice spec; new hires can't trace why you picked Postgres; vendor changes leak into prose nobody updates.

BBC says: memory is the contract. Decisions live as markdown files with a strict schema. Cross-layer changes go through a proposal queue (propose → review → accept/reject). Vendors are abstracted behind named roles (db-provider, llm-provider) so you can swap them without rewriting prose. The audit trail is forever.

The product is the protocol + dashboard + MCP server — your tenant content (your team's actual decisions, voice, bindings) plugs in via a separate repo or via Supabase rows. Skeleton + slot.

2. The shape: skeleton + slot

github.com/ZethT/bbc (public, AGPL-3.0) apps/dashboard // Next.js dashboard UI apps/mcp-server // Model Context Protocol bridge for agents packages/store // typed storage interface scripts/ // bash protocol scripts manager/ // Manager rules + queue-review agent memory/_schema.md // the contract memory/ops/provider-roles/ // db-provider, llm-provider, … memory/ops/providers/example-* // adapter shape (placeholders) templates/initial-tenant/ // bare forkable skeleton examples/example-tenant/ // runnable Acme Co demo docs/ // this file lives here ↕ plugged in via BBC_REPO env var (file-mode) ↕ or via Supabase rows (DB-mode, multi-tenant SaaS) your-tenant-repo (private — your call) CLAUDE.md // your Main precedence rules memory/ // your decisions, voice, glossary, vendors ops/bindings.yaml // your role → vendor table ops/providers/*.yaml // your real adapter declarations distribution/<leaf>/ // per-workstream context queue/ // live + archived proposals _log/operations.jsonl // your audit trail

BBC ships generic. Your tenant ships specific. The dashboard reads BBC_REPO at request time and renders whatever tenant is plugged in.

3. Run it in 60 seconds

git clone https://github.com/ZethT/bbc
cd bbc
pnpm install
pnpm --filter @bbc/dashboard dev
# → http://localhost:3000  (against examples/example-tenant — Acme Co demo)

Or via Docker (after copying .env.example → .env with your Supabase keys):

docker compose up --build
You'll see Acme Co — a fictional 3-person team with sample decisions, bindings, audit log, queue items. It's not real; it's the demo. Click around /queue, /log, /bindings, /team to see the surfaces.

4. The protocol in 3 paragraphs

Three layers of trust. Main > Manager > Distribution. Main holds principles (locked from below). Manager defines product workflow + queue review. Distribution is per-workstream context (one folder per repo/team you govern). Lower layers can specialize upper rules; they cannot override. Conflict → Main wins.

Memory has a schema. Every file under memory/ has YAML frontmatter (id, type, scope, layer, source, owning_layer, status). The schema in memory/_schema.md is the contract. Storage is a binding: markdown files in self-host file-mode, RLS-gated rows in the multi-tenant SaaS. The shape stays the same.

Changes go through the queue. A leaf proposes. The Manager reviews. An admin accepts or rejects. Both outcomes stay forever as the audit trail. The dashboard surfaces this at /queue; the bash equivalents are scripts/{propose,accept,reject}.sh.

5. Your first tenant

Two paths. Pick by audience:

Path What you get Who it's for
Fork the skeleton A bare tenant repo (CLAUDE.md, schema, sample queue) you can flesh out from scratch. You want full control + git history of your tenant content.
Self-serve sign up Land on /auth/self-serve, pick a tenant name, get a fresh tenant with you as admin (gated by BBC_SIGNUP_MODE=open). Trying out the SaaS path; want everything in one place.

Fork path

mkdir ~/Documents/GitHub/your-tenant
cd ~/Documents/GitHub/your-tenant
cp -R /path/to/bbc/templates/initial-tenant/. .
git init && git add -A && git commit -m "Bootstrap from BBC initial-tenant template"

# Edit CLAUDE.md, memory/decisions/0001-*.md, memory/people/team.md, memory/ops/bindings.yaml — make them YOURS.

# Run BBC against your new tenant:
cd /path/to/bbc
BBC_REPO=~/Documents/GitHub/your-tenant pnpm --filter @bbc/dashboard dev

Self-serve path

echo "BBC_SIGNUP_MODE=open" >> apps/dashboard/.env.local
pnpm --filter @bbc/dashboard dev
# Visit /auth/signin → "Create my own tenant" link → /auth/self-serve
Self-serve needs Supabase to send a confirmation email. If you haven't wired SMTP (Resend / Postmark / SES), you'll create a tenant but won't be able to sign in. For local dev, manually confirm via Supabase Studio → Auth → Users.

6. Wiring agents (MCP)

BBC ships an MCP server at apps/mcp-server/ so agents (Claude Desktop, Cursor, your custom integrations) can read+write a tenant's BBC instance. Per-tenant API tokens, RLS-gated under the hood.

Issue a token at /api-keys (admin role required). The plaintext is shown ONCE in a panel — copy it before navigating away. Format: bbc_<key_id>.<secret>.

Pass on the wire as a standard bearer token:

Authorization: Bearer bbc_<key_id>.<secret>

Tools available today (read-only; write tools are a Phase 6.1 follow-up):

  • read_memory — fetch a memory file by path
  • list_queue — list queue items by status
  • read_log — recent operations_log entries

Wire-up details, transport config, security model: see apps/mcp-server/README.md.
